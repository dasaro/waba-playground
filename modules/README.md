# Playground Modules

This directory contains the browser runtime for the static WABA playground.

## Runtime Split

- [clingo-manager.js](/Users/fdasaro/Desktop/WABA-claude/ABA-variants/waba-playground/modules/clingo-manager.js)
  - resolves semiring family + polarity
  - injects default policy, monoid, optimization, and budget modules
  - initializes the Clingo worker with the correct path-relative WASM URL
  - serializes browser-side solver calls so graph recomputation and semantics runs cannot race
  - runs exact browser-side `preferred`
- [graph-manager.js](/Users/fdasaro/Desktop/WABA-claude/ABA-variants/waba-playground/modules/graph-manager.js)
  - keeps the existing vis.js graph architecture
  - builds graph data from the framework plus the selected semiring/default policy
- [output-manager.js](/Users/fdasaro/Desktop/WABA-claude/ABA-variants/waba-playground/modules/output-manager.js)
  - parses answer sets
  - displays extension summaries
  - groups threshold rankings when unbounded exploration is enabled
- [metrics-manager.js](/Users/fdasaro/Desktop/WABA-claude/ABA-variants/waba-playground/modules/metrics-manager.js)
  - groups identical `in/1` sets into unique decision alternatives
  - ranks assumptions with a Borda-style decision score over extension levels
  - reports robustness and leverage for “best assumption / best course of action” analysis
- [wasm-config.js](/Users/fdasaro/Desktop/WABA-claude/ABA-variants/waba-playground/modules/wasm-config.js)
  - resolves `dist/` assets relative to the current page URL

## Generated Bundle

[waba-modules.js](/Users/fdasaro/Desktop/WABA-claude/ABA-variants/waba-playground/waba-modules.js) is generated by [scripts/sync-modules.js](/Users/fdasaro/Desktop/WABA-claude/ABA-variants/waba-playground/scripts/sync-modules.js).

Its schema is:

```js
wabaModules = {
  core,
  semiring,
  defaults,
  monoid,
  optimize,
  constraint,
  filter,
  semantics,
  examples,
  metadata
}
```

`metadata` carries the canonical split used by the UI and runtime:

- semiring families and polarities
- canonical family/polarity -> module resolution
- alias labels (`arctic`, `bottleneck_cost`)
- supported semantics
- supported bounded presets

## Exact Preferred

The playground stays plain-`clingo` and frontend-only. Exact `preferred` is therefore implemented as:

1. enumerate `complete` candidates with the selected algebra/budget configuration
2. run `subset_maximal_filter.lp` on generated `candidate/1` and `member/2` facts
3. optionally apply numeric post-filtering if the user asked for `optN`

This mirrors the mature WABA surface without requiring `asprin` or a backend wrapper.
